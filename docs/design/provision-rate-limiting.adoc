= Placement Queue for Load Protection
:toc:
:toc-placement: preamble

Protect shared clusters from provisioning storms by rate-limiting how many new placements can be scheduled within a configurable time window. When all clusters are at capacity, placements are queued and processed FIFO.

== Problem

When many placements start simultaneously on a shared cluster, the cluster cannot handle the load ‚Äî not because of total capacity, but because of concurrent provisioning pressure. We need to spread provisioning over time.

== Solution Overview

Two new configuration fields per cluster, stored in a new `settings` JSONB column:

- `provision_rate_limit` ‚Äî maximum number of new placements allowed within the time window
- `provision_rate_window` ‚Äî the time window (e.g. `10m`, `1h`)

If all candidate clusters have exhausted their rate limit, the placement enters a `"queued"` status and is processed later by a background goroutine (FIFO order).

=== Introducing the `settings` column

The existing `data` JSONB column (`ClusterData` struct) holds **system-managed internal state** ‚Äî SA token rotation timestamps, connection status, placement counts. The upsert handler overwrites `Data` with the existing value on every update to protect these plumbing fields.

A new `settings` JSONB column (`ClusterSettings` struct) cleanly separates **user-configurable settings** from internal plumbing:

[cols="1,2,2"]
|===
| Column | Purpose | Update behavior

| `data`
| System-managed internal state (rotation timestamps, connection status, placement counts)
| Preserved on update ‚Äî handler overwrites with existing values

| `settings`
| User-configurable settings (rate limits, future settings)
| Taken from request ‚Äî user controls these values
|===

*Why* ? Having a JSONB field is more flexible and doesn‚Äôt require a DB migration everytime we want to add a field. We‚Äôll eventually migrate top-level settings like max_placements under that new settings

== Configuration

Set via the cluster configuration PUT endpoint or `sandbox-cli cluster onboard`:

[source,json]
----
{
  "name": "my-cluster",
  "annotations": {"cloud": "aws"},
  "max_placements": 30,
  "settings": {
    "provision_rate_limit": 50,
    "provision_rate_window": "10m"
  }
}
----

[source,shell]
----
sandbox-cli cluster onboard \
  --name my-cluster \
  --annotations '{"cloud":"aws"}' \
  --provision-rate-limit 50 \
  --provision-rate-window 10m
----

Both rate-limit fields must be set together. When absent or null, no rate limiting is applied (backward compatible).

== Scheduling Flow

[source]
----
Placement request arrives
‚îÇ
‚îú‚îÄ For each resource in the placement:
‚îÇ   ‚îú‚îÄ Find candidate clusters (existing: annotations, max_placements)
‚îÇ   ‚îÇ
‚îÇ   ‚îú‚îÄ For each candidate, check provision rate
‚îÇ   ‚îÇ   ‚îî‚îÄ COUNT resources WHERE ocp_cluster = $1
‚îÇ   ‚îÇ       AND created_at > now() - provision_rate_window
‚îÇ   ‚îÇ   ‚îî‚îÄ If recent_count >= provision_rate_limit ‚Üí skip cluster
‚îÇ   ‚îÇ
‚îÇ   ‚îú‚îÄ Cluster available ‚Üí create resource with status: "initializing"
‚îÇ   ‚îÇ   ‚îî‚îÄ Start async provisioning goroutine
‚îÇ   ‚îÇ
‚îÇ   ‚îî‚îÄ ALL clusters rate-limited
‚îÇ       ‚îî‚îÄ Create resource with status: "queued"
‚îÇ           ‚îî‚îÄ No async provisioning goroutine ‚Äî resource waits in queue
‚îÇ
‚îú‚îÄ Cluster relation handling (same_cluster_as, different_cluster_from):
‚îÇ   ‚îú‚îÄ Resources WITH cluster relations skip the synchronous rate-limit
‚îÇ   ‚îÇ   pre-check (scheduling is deferred to the async task)
‚îÇ   ‚îú‚îÄ If a referenced sibling resource is "queued", the dependent
‚îÇ   ‚îÇ   resource is also created as "queued" (it can't resolve yet)
‚îÇ   ‚îî‚îÄ The queue processor dequeues them in dependency order
‚îÇ
‚îî‚îÄ After all resources processed:
    ‚îú‚îÄ Any resource "queued" ‚Üí set placement status to "queued"
    ‚îî‚îÄ All resources "initializing" ‚Üí normal flow
----

=== Resource-Level Queuing (Why Not Placement-Level)

The initial design queued entire placements. This breaks with multi-resource placements:

[source]
----
Placement with 2 resources (A targets cluster X, B targets cluster Y):
  1. Resource A: Request() ‚Üí success, resource created, async provisioning starts
  2. Resource B: Request() ‚Üí ErrRateLimited (cluster Y at limit)
  3. ‚ùå WRONG: Queue entire placement ‚Üí Resource A is orphaned
  4. ‚ùå WRONG: Dequeue later ‚Üí tries to re-create A ‚Üí conflict
----

Resource-level queuing solves this:

[source]
----
  1. Resource A: Request() ‚Üí cluster X available ‚Üí "initializing" (provisions normally)
  2. Resource B: Request() ‚Üí cluster Y rate-limited ‚Üí "queued" (waits)
  3. Placement status = "queued" (because B is queued)
  4. Queue processor dequeues B when cluster Y has capacity
  5. B provisions ‚Üí all resources done ‚Üí placement status = "success"
----

Each resource is independently queued or provisioned. The placement status is derived from the aggregate state of its resources.

== Queue Processing

A background goroutine polls for queued *resources*:

- **Interval:** 30 seconds
- **Order:** `created_at ASC` (FIFO)
- **Action:** For each queued resource:
  1. Check if cluster relations are satisfied (referenced siblings must be provisioned)
  2. Re-check rate limits on candidate clusters
  3. Rate limit available ‚Üí trigger provisioning (status ‚Üí `"initializing"`)
  4. Still rate-limited or dependency not met ‚Üí leave as `"queued"`
- **After provisioning a resource:** Check if all resources for the placement are done. If so, the placement status is updated by the normal `LoadActiveResources` flow.
- **Startup:** Launched alongside existing background goroutines in `main.go`
- **Concurrency:** Uses `pg_try_advisory_xact_lock` so only one pod processes the queue at a time.

== Resource and Placement Status: `"queued"`

New status added to both enums:

- `resource_status_enum`: `queued` added before `initializing`
- `placement_status_enum`: `queued` added before `initializing` (existing migration 024)

=== Resource lifecycle

[source]
----
Resource created:
  ‚îú‚îÄ Rate limit OK    ‚Üí "initializing" ‚Üí "scheduling" ‚Üí "success"
  ‚îÇ                                                    ‚Üí "error"
  ‚îî‚îÄ Rate limited     ‚Üí "queued" ‚Üí (dequeued) ‚Üí "initializing" ‚Üí ...
----

- Resource is created in DB with `status = "queued"`
- Provisioning parameters (cloud_selector, quota, limit_range, cluster_relation, etc.) are stored in `resource_data` JSONB
- No async provisioning goroutine runs until the resource is dequeued

=== Placement lifecycle

[source]
----
Placement:
  ‚îú‚îÄ All resources "initializing" ‚Üí normal flow ‚Üí "success"
  ‚îú‚îÄ Any resource "queued"        ‚Üí "queued" ‚Üí (resources dequeue) ‚Üí "success"
  ‚îî‚îÄ                                                                ‚Üí "error"
----

- Placement `status = "queued"` is set by the handler when any OcpSandbox resource is queued
- Anarchy sees `"queued"` and reports "in queue" to the user
- HTTP response is the same *202 Accepted* as any new placement
- `LoadActiveResources` skips status update when placement is `"queued"` (status is managed by the queue processor)

== Dry-Run

The dry-run endpoint shows per-cluster rate-limit state via a new `cluster_details` field.

=== Capacity available (immediate scheduling)

[source,json]
----
{
  "kind": "OcpSandbox",
  "available": true,
  "message": "Matching OCP shared clusters found",
  "schedulable_cluster_count": 3,
  "schedulable_cluster_names": ["cluster1", "cluster2", "cluster3"],
  "cluster_details": [
    {"name": "cluster1", "available_slots": 50},
    {"name": "cluster2", "available_slots": 8},
    {"name": "cluster3", "available_slots": 35}
  ]
}
----

=== All clusters at rate limit (would be queued)

[source,json]
----
{
  "kind": "OcpSandbox",
  "available": true,
  "message": "Matching OCP shared clusters found (placement would be queued)",
  "schedulable_cluster_count": 3,
  "schedulable_cluster_names": ["cluster1", "cluster2", "cluster3"],
  "cluster_details": [
    {"name": "cluster1", "available_slots": 0},
    {"name": "cluster2", "available_slots": 0},
    {"name": "cluster3", "available_slots": 0}
  ],
  "queued": true,
  "queue_position": 5
}
----

- `available` is `true` ‚Äî the placement _will_ happen, just not immediately
- `available_slots` = `provision_rate_limit - recent_provisions_in_window`
  * 0 = cluster is at its rate limit
  * Equals `provision_rate_limit` during quiet periods
- Clusters without rate limiting configured omit `available_slots` (unlimited)
- `queue_position` = number of existing queued placements ahead (for the same cloud selector)
- `available` is `false` ‚Äî no matching cluster, or all cluster at max capacity (max_placements)

== API Changes

=== Cluster Configuration

`PUT /api/v1/ocp-shared-cluster-configurations/{name}`

New `settings` JSONB field (replaces per-field approach):

[cols="2,1,3"]
|===
| Field | Type | Description

| `settings.provision_rate_limit`
| integer (nullable)
| Max new placements within the rate window. Must be > 0.

| `settings.provision_rate_window`
| string (nullable)
| Duration string (e.g. `10m`, `1h`, `30s`). Must be set with `provision_rate_limit`.
|===

=== Placement Creation

`POST /api/v1/placements`

No change to HTTP status codes ‚Äî new placements already return *202 Accepted*. The only difference is the placement `status` field:

[cols="1,3"]
|===
| Placement status | Meaning

| `"initializing"`
| Placement scheduled immediately (existing behavior)

| `"queued"`
| All candidate clusters at rate limit ‚Äî will be processed later (FIFO)
|===

The existing *200 OK* response for identical duplicate requests is unchanged.

=== Dry-Run

`POST /api/v1/placements/dry-run`

New fields in `ResourceDryRunResult`:

[cols="2,1,3"]
|===
| Field | Type | Description

| `cluster_details`
| array (optional)
| Per-cluster rate-limit info. Only present for OcpSandbox resources.

| `cluster_details[].name`
| string
| Cluster name

| `cluster_details[].available_slots`
| integer (optional)
| Remaining capacity in the rate window. Omitted if no rate limit configured.

| `queued`
| boolean (optional)
| `true` if the placement would be queued. Only present when all clusters are rate-limited.

| `queue_position`
| integer (optional)
| Position in the queue. Only present when `queued` is `true`.
|===
