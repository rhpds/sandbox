= Placement Queue for Load Protection
:toc:
:toc-placement: preamble

Protect shared clusters from provisioning storms by rate-limiting how many new placements can be scheduled within a configurable time window. When all clusters are at capacity, placements are queued and processed FIFO.

== Problem

When many placements start simultaneously on a shared cluster, the cluster cannot handle the load — not because of total capacity, but because of concurrent provisioning pressure. We need to spread provisioning over time.

== Solution Overview

Two new configuration fields per cluster, stored in a new `settings` JSONB column:

- `provision_rate_limit` — maximum number of new placements allowed within the time window
- `provision_rate_window` — the time window (e.g. `10m`, `1h`)

If all candidate clusters have exhausted their rate limit, the placement enters a `"queued"` status and is processed later by a background goroutine (FIFO order).

=== Introducing the `settings` column

The existing `data` JSONB column (`ClusterData` struct) holds **system-managed internal state** — SA token rotation timestamps, connection status, placement counts. The upsert handler overwrites `Data` with the existing value on every update to protect these plumbing fields.

A new `settings` JSONB column (`ClusterSettings` struct) cleanly separates **user-configurable settings** from internal plumbing:

[cols="1,2,2"]
|===
| Column | Purpose | Update behavior

| `data`
| System-managed internal state (rotation timestamps, connection status, placement counts)
| Preserved on update — handler overwrites with existing values

| `settings`
| User-configurable settings (rate limits, future settings)
| Taken from request — user controls these values
|===

*Why* ? Having a JSONB field is more flexible and doesn't require a DB migration everytime we want to add a field. We'll eventually migrate top-level settings like max_placements under that new settings

== Configuration

Set via the cluster configuration PUT endpoint or `sandbox-cli cluster onboard`:

[source,json]
----
{
  "name": "my-cluster",
  "annotations": {"cloud": "aws"},
  "max_placements": 30,
  "settings": {
    "provision_rate_limit": 50,
    "provision_rate_window": "10m"
  }
}
----

[source,shell]
----
sandbox-cli cluster onboard \
  --name my-cluster \
  --annotations '{"cloud":"aws"}' \
  --provision-rate-limit 50 \
  --provision-rate-window 10m
----

Both rate-limit fields must be set together. When absent or null, no rate limiting is applied (backward compatible).

== Scheduling Flow

[source]
----
Placement request arrives
│
├─ For each resource in the placement:
│   ├─ Find candidate clusters (existing: annotations, max_placements)
│   │
│   ├─ Create resource with status: "initializing" (saved to DB)
│   │
│   ├─ Synchronous slot reservation (resources WITHOUT cluster relations):
│   │   └─ Loop through candidate clusters calling CheckAndReserveSlot()
│   │       which uses pg_advisory_xact_lock to serialize concurrent checks
│   │   └─ If one cluster has capacity → slot is reserved, proceed to async
│   │   └─ If ALL clusters rate-limited → convert resource to "queued"
│   │       └─ No async provisioning goroutine — resource waits in queue
│   │
│   ├─ Async provisioning goroutine (if slot was reserved):
│   │   └─ Pre-reserved cluster is tried first (skips rate-limit re-check)
│   │   └─ If pre-reserved cluster fails (connectivity, etc.) → fallback to
│   │       other candidates with normal CheckAndReserveSlot() checks
│   │
│   └─ Resource ends up with status: "initializing", "scheduling", or "queued"
│
├─ Cluster relation handling (same_cluster_as, different_cluster_from):
│   ├─ Resources WITH cluster relations skip the synchronous slot
│   │   reservation (candidate clusters can't be resolved until siblings
│   │   provision). Rate-limit checks happen in the async goroutine via
│   │   CheckAndReserveSlot() after cluster relations are resolved.
│   ├─ If a referenced sibling resource is "queued", the dependent
│   │   resource is also created as "queued" (it can't resolve yet)
│   └─ The queue processor dequeues them in dependency order
│
└─ After all resources processed:
    ├─ Any resource "queued" → set placement status to "queued"
    └─ All resources "initializing" → normal flow
----

=== Concurrency Safety (Advisory Locks)

A naive implementation checks the rate limit and reserves the slot as two separate operations. Under concurrent burst load, multiple requests can all see count=0 and all pass through, exceeding the rate limit (TOCTOU race).

The solution uses PostgreSQL advisory locks (`pg_advisory_xact_lock`) keyed on a hash of the cluster name:

1. **Synchronous slot reservation (`CheckAndReserveSlot`):** After the resource is saved to the database, the `Request()` function synchronously loops through candidate clusters calling `CheckAndReserveSlot()`. Each call acquires an advisory lock for one cluster, checks the count, and if under the limit, reserves the slot (updates `resource_data->>'ocp_cluster'`) — all within the same transaction. The lock is released on commit/rollback.

2. **Why synchronous?** The resource must exist in the database _before_ the rate-limit count check runs. This prevents the TOCTOU race where concurrent requests all see count=0 and all pass through. By saving first and then checking under an advisory lock, each concurrent request sees the correct count of existing resources.

3. **Pre-reserved cluster in async goroutine:** When a slot is reserved synchronously, the cluster name is passed to the async provisioning goroutine. The goroutine skips `CheckAndReserveSlot` for this cluster and uses the pre-reserved slot directly. If the pre-reserved cluster fails (connectivity, etc.), the goroutine falls back to other candidates with normal rate-limit checks.

These locks are lightweight (in-memory, no disk I/O) and scoped to the transaction lifetime.

=== Dry-Run (No Locks)

The dry-run endpoint uses the plain `IsRateLimited()` method, which is a simple `SELECT count(*)` query with no advisory locks. Dry-run is read-only and must stay fast — it never reserves slots or modifies state.

=== Resource-Level Queuing (Why Not Placement-Level)

The initial design queued entire placements. This breaks with multi-resource placements:

[source]
----
Placement with 2 resources (A targets cluster X, B targets cluster Y):
  1. Resource A: Request() → success, resource created, async provisioning starts
  2. Resource B: Request() → ErrRateLimited (cluster Y at limit)
  3. ❌ WRONG: Queue entire placement → Resource A is orphaned
  4. ❌ WRONG: Dequeue later → tries to re-create A → conflict
----

Resource-level queuing solves this:

[source]
----
  1. Resource A: Request() → cluster X available → "initializing" (provisions normally)
  2. Resource B: Request() → cluster Y rate-limited → "queued" (waits)
  3. Placement status = "queued" (because B is queued)
  4. Queue processor dequeues B when cluster Y has capacity
  5. B provisions → all resources done → placement status = "success"
----

Each resource is independently queued or provisioned. The placement status is derived from the aggregate state of its resources.

== Queue Processing

A background goroutine polls for queued *resources*:

- **Interval:** 30 seconds by default, configurable via `QUEUE_POLL_INTERVAL` env var (e.g. `5s` for testing)
- **Parallelism:** 1 processor by default, configurable via `QUEUE_PROCESSORS` env var (e.g. `20` for stress testing)
- **Order:** `created_at ASC` (FIFO)
- **Action:** For each queued resource:
  1. Check if cluster relations are satisfied (referenced siblings must be provisioned)
  2. Re-check rate limits on candidate clusters (using `IsRateLimited()`, read-only)
  3. Rate limit available → trigger provisioning (status → `"initializing"`)
  4. Still rate-limited or dependency not met → leave as `"queued"`
- **After provisioning a resource:** Check if all resources for the placement are done. If so, the placement status is updated by the normal `LoadActiveResources` flow.
- **Startup:** Launched alongside existing background goroutines in `main.go`
- **Concurrency:** Uses `pg_try_advisory_xact_lock` so only one processor (across all pods) processes the queue at a time. Multiple `QUEUE_PROCESSORS` goroutines compete for the same lock — only the winner runs, the rest skip.

== Resource and Placement Status: `"queued"`

New status added to both enums:

- `resource_status_enum`: `queued` added before `initializing` (migration 025)
- `placement_status_enum`: `queued` added before `initializing` (migration 024)

=== Resource lifecycle

[source]
----
Resource created in DB with status "initializing":
  ├─ Synchronous slot reservation succeeds
  │   → "scheduling" → "success"
  │                   → "error"
  └─ All clusters rate-limited (synchronous check)
      → converted to "queued" (status + queued_params saved)
      → (queue processor dequeues) → re-enters Request() → "initializing" → ...
----

- Resource is always created first as `"initializing"`, then converted to `"queued"` if all clusters are rate-limited. This ensures the resource exists in the DB before rate-limit count queries run, preventing TOCTOU races.
- Provisioning parameters (cloud_selector, quota, limit_range, cluster_relation, etc.) are stored in `resource_data` JSONB as `queued_params`
- No async provisioning goroutine runs until the resource is dequeued
- Queued resources have no namespace — `Release()` skips the namespace check for `status = "queued"`
- Rate-limit count queries exclude resources marked `to_cleanup = true` (deleted resources don't count against the limit)

=== Placement lifecycle

[source]
----
Placement:
  ├─ All resources "initializing" → normal flow → "success"
  ├─ Any resource "queued"        → "queued" → (resources dequeue) → "success"
  └─                                                                → "error"
----

- Placement `status = "queued"` is set by the handler when any OcpSandbox resource is queued
- Anarchy sees `"queued"` and reports "in queue" to the user
- HTTP response is the same *202 Accepted* as any new placement
- `LoadActiveResources` skips status update when placement is `"queued"` (status is managed by the queue processor)

== Dry-Run

The dry-run endpoint (`POST /api/v1/placements/dry-run`) shows per-cluster rate-limit state. It is read-only — no locks, no state changes, fast response.

=== Capacity available (immediate scheduling)

[source,json]
----
{
  "kind": "OcpSandbox",
  "available": true,
  "message": "Matching OCP shared clusters found",
  "schedulable_cluster_count": 3,
  "schedulable_cluster_names": ["cluster1", "cluster2", "cluster3"],
  "cluster_details": [
    {"name": "cluster1", "available_slots": 50},
    {"name": "cluster2", "available_slots": 8},
    {"name": "cluster3", "available_slots": 35}
  ]
}
----

=== All clusters at rate limit (would be queued)

[source,json]
----
{
  "kind": "OcpSandbox",
  "available": true,
  "message": "Matching OCP shared clusters found (placement would be queued)",
  "schedulable_cluster_count": 3,
  "schedulable_cluster_names": ["cluster1", "cluster2", "cluster3"],
  "cluster_details": [
    {"name": "cluster1", "available_slots": 0},
    {"name": "cluster2", "available_slots": 0},
    {"name": "cluster3", "available_slots": 0}
  ],
  "queued": true,
  "queue_position": 5
}
----

- `available` is `true` — the placement _will_ happen, just not immediately
- `available_slots` = `provision_rate_limit - recent_provisions_in_window`
  * 0 = cluster is at its rate limit
  * Equals `provision_rate_limit` during quiet periods
- Clusters without rate limiting configured omit `available_slots` (unlimited)
- `queue_position` = number of existing queued placements ahead (for the same cloud selector)
- `available` is `false` — no matching cluster, or all cluster at max capacity (max_placements)

== API Changes

=== Cluster Configuration

`PUT /api/v1/ocp-shared-cluster-configurations/{name}`

New `settings` JSONB field (replaces per-field approach):

[cols="2,1,3"]
|===
| Field | Type | Description

| `settings.provision_rate_limit`
| integer (nullable)
| Max new placements within the rate window. Must be > 0.

| `settings.provision_rate_window`
| string (nullable)
| Duration string (e.g. `10m`, `1h`, `30s`). Must be set with `provision_rate_limit`.
|===

=== Placement Creation

`POST /api/v1/placements`

No change to HTTP status codes — new placements already return *202 Accepted*. The only difference is the placement `status` field:

[cols="1,3"]
|===
| Placement status | Meaning

| `"initializing"`
| Placement scheduled immediately (existing behavior)

| `"queued"`
| All candidate clusters at rate limit — will be processed later (FIFO)
|===

The existing *200 OK* response for identical duplicate requests is unchanged.

=== Dry-Run

`POST /api/v1/placements/dry-run`

New fields in `ResourceDryRunResult`:

[cols="2,1,3"]
|===
| Field | Type | Description

| `cluster_details`
| array (optional)
| Per-cluster rate-limit info. Only present for OcpSandbox resources.

| `cluster_details[].name`
| string
| Cluster name

| `cluster_details[].available_slots`
| integer (optional)
| Remaining capacity in the rate window. Omitted if no rate limit configured.

| `queued`
| boolean (optional)
| `true` if the placement would be queued. Only present when all clusters are rate-limited.

| `queue_position`
| integer (optional)
| Position in the queue. Only present when `queued` is `true`.
|===
